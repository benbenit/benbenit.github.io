什么是进程

  • 在本套课程里面，进程就仅仅提供程序运行所需要的资源，比如数据，代码。至于程序运行的那种状态，和进程没有关系。而使用这些资源的”人”，我们叫做线程

  • 从上面可以看到，地址是从0001 0000开始的，是因为前面的64K是空白的，正常情况下这64K是没人使用的
  • 从上面可以看到，一个进程并不只是有一个文件组成的，而是由一堆文件组成的，点E可以从另外一个角度看这个进程到底提供了哪些所谓的代码，又提供了哪些代码
  • 从上面看，当前的进程是由一个一个模块组成的，每个模块可以看成是一个零件。进程的职责是提供代码和数据，这些代码和数据并不在一个”零件”里，而是分散到不同的”零件”里，也就是一个一个模块里。每一个dll都是我们说的模块，每一个dll都提供代码和数据
  • 所有的进程里的模块都遵守着相同的格式，我们称为pe文件结构，基础四里面专门讲解PE文件结构。现阶段只需要了解，每一个进程都是由一些PE文件组成的，每一个PE文件都占一段内存

  • 在创建完线程之后，并不能直接启动线程，而是还要映射一些dll

  • 如果仅仅是做应用层的开发的话，会用api就行了，但是要做底层相关的工作，这些东西是必须要掌握的，比如说如果了解了上面的步骤，就可以做些手脚。比如在第5步的时候，正常情况下应该映射A.exe的dll，但是现在把它换成B.exe，那启动的就成了B.exe
  • 再举一个例子，一个游戏程序，你写一个外挂，就相当于写一个dll，把它放到游戏的进程空间里面去执行，达到自己的目的。通常游戏都有检测外挂的机制，通常是写到dll里，如果熟悉进程创建的过程，就可以让检测dll进入进程空间之前，把我们的dll放到进程空间里，这样就避过检测了
  • 当然上面也不是通杀，在技术的领域，拼的永远是基础，看谁对细节理解得到位

如何管理内核对象

  • 在创建内核对象的目的是为了使用他们，那该怎么使用，怎么管理？
  • 上面这张图简单表示，就是我在一个进程里面又创建了一个进程，创建了一个线程，创建了一个文件，以及事件。他们在0环都有对应的内核结构体，那怎么管理，怎么使用这些内核结构体?
  • 最简单的方案就是在创建完内核结构体之后，把内核结构体的地址返回回去，等想使用的时候就可以拿着这个地址找到内核结构体，但是这样做会有问题。因为这些内核结构体是在0环创建的，那就意味着这些内核结构体的内存地址是大于80的，比如A结构体地址返回给应用层了，但是应用层把这个内存地址改了，再次访问的时候就会造成内存不可访问，访问内核如果地址给错了，会直接蓝屏。
  • windows为了避免这种问题，就创建了句柄表，用句柄表来管理。这样就相当于在内核层和用户层之间建立了一层屏障，让我们的应用程序无法直接访问内核层的这些内核对象
  • 句柄表并不是说每个内核对象都有，只有进程的内核对象才有句柄表，这张表存的实际上是一种映射关系，我们得到的所谓的句柄的值，实际上是张表里的一个索引

多进程共享内核对象

  • 我们知道0环是所有进程共享的，而内核结构体也是在0环，这就意味着这些内核结构体是能够跨进程共享的
  • 通过OpenProcess可以打开一个进程
  • B进程也是通过自己的句柄表来操作A进程的内核对象，他和A进程的句柄表没有任何关系，是一人一张私有的表
  • 上图中的2代表了一个计数器，在A进程里面创建了一个内核对象，他的计数器就是1，如果有B进程又打开了这个内核对象，他的计数器就是2，再有人打开就是3
  • 使用CloseHandle关闭句柄，传进来的参数就是句柄的值，很多人以为，这个函数会把这个内核对象关掉，实际上不是。他仅仅是把计数器的值减1
  • 当计数器变成0，变成没有任何人指向的内核对象，这个时候，就说这个内核对象”死”了
  • 任何一个内核对象都有类似的特征，但是有例外，线程
  • 关闭线程内核对象，首先要先关闭线程，然后这个计数器为0，否则这个内核对象是不会”死”的
  • 上面我们已经了解了一种两个进程共享内核对象的方式了，windows除了上面的这种方式之外，还提供了另外一种共享内核对象的方式

句柄是否可以被继承

  • 之前讲过使用关闭句柄的函数来判断是不是内核对象，现在有另一种方法，看他们的创建函数，是不是有安全描述符
  • 安全描述符对我们来说没什么用，它里面主要有一个成员控制那个成员能访问，指明这个内核对象能给谁用不能给谁用，谁能关闭谁不能关闭等等，通常我们写代码的时候不用去关注他，默认的设置和父进程是一样的
  • 真正有价值的是bInheritHandle，这个参数说明当前的内核对象是否允许继承
  • 之前画的图里面，句柄表有两列，但实际上不是，至少有3列，还有一列是说明这个内核对象能否被继承，能继承是1，不能继承是0
  • 最最关键的一点，比如说在一个进程里面，创建了很多内核结构体，就会有一张记录的句柄表，子进程是有机会把这张句柄表里的值直接继承过去的。如果是两个平行的进程，使用openprocess，而如果是子父进程的话，在创建子进程的话，在参数地方写成true，就意味着子进程能够得到父进程的句柄表，当然也不是全部得到，会继承安全描述符是1，即继承允许继承的句柄表，把不允许继承的填上0，所以这也是一种共享内核的方案

进程相关API

  • 之间讲过句柄了，返回来的还有两个没有说，就是所谓的id
  • 句柄是每一个进程都有的私有的一张表，存储了了打开的，创建的所有的内核对象，除了每个进程之外，操作系统也有一份表，这张表不像进程私有的句柄表，而是全局只有一份，叫做全局句柄表，在操作系统的全局句柄表里面，别的先不管，里面至少包含了所有正在运行的进程和线程
  • 表的结构和进程私有的那张句柄表没什么区别
  • 回过头来看，为什么有进程id，线程id的概念，进程id，线程id就是操作系统里面，全局的句柄表里面的一个编号，索引，这个编号是唯一的，但是唯一并不代表不变，某个进程死了，这个编号就可能给其他进程使用
  • 这意味着句柄只在当前进程里面有意义，出了当前进程就没有意义，但是全局句柄表里的所以不一样，出了当前进程，在别的进程里面仍然有意义

什么是线程

  • 线程是附属在进程上的执行实体，是代码的执行流程
  • 一个进程可以包含多个线程，但一个进程至少包含一个线程
  • 比如说一个进程有3个线程，意味着什么？意味着在这个进程中，有3段代码在执行，并不一定是同时执行，比如说单核的，某一个时刻只能有一段代码在执行，但是在代码之间切换的速度非常快，给我们的感觉就像同时在执行
  • 单核情况下，并不存在真正的多线程。代码执行的时候需要寄存器，一个CPU只有一套寄存器，所以不可能多个线程同时跑。但是如果是多核的，就可以多个线程，多个代码段同时执行
  • 返回值是一个句柄，这个句柄的作用就是，后面如果想控制这个线程的话，就通过这个句柄来控制

临界区

线程安全问题

  • 每个线程有自己的栈，而局部变量是存储在栈中的，这意味着每个线程都有一份自己的局部变量，如果线程仅仅使用局部变量，那就不存在线程安全问题
  • 如果多线程共用一个全局变量，那就可能存在线程安全问题。线程安全问题出现的前提是有共用的全局变量，而且对这个全局变量不仅仅是只读的，要修改全局变量，这个时候就可能出现线程安全问题
  • 临界资源，指的是一次只能是一个线程使用的资源。全局变量出错，就是因为没有把它变成临界资源
  • 临界区，指的是对临界资源访问的这段代码
  • 我们要保证对全局变量的访问是安全的，就需要自己构建一段临界区，这段临界区可以自己写代码实现，也可以使用windows提供的API来实现，中级班会指导自己写代码实现，现在先说windows的方式
  • windows的实现方式，设置一个全局变量，我们给这个全局变量叫令牌，临界区的代码想要访问全局变量，首先要获取令牌，这个令牌是1或者0，拿到之后把令牌置为0，表示被人拿到了。在拿到之后，其他线程想要访问全局变量的话，也要先去拿这个令牌，如果令牌为0，就不能访问
  • 临界区的实现在这个阶段先不讨论。windows实现的基本思想就是，把判断和设置令牌的操作设置为原子操作，这样就不出现两个线程同时进行判断的情况

  • 互斥体还有一个作用，比如为了防止应用多开，不允许运行多份，可以通过互斥体控制。当然也可以通过文件，通过共享内存来控制

什么是消息

  • 当我们点击鼠标，或者当我们按下键盘的时候，操作系统都要把这些动作记录下来，存储到一个结构体中，这个结构体就是消息

消息队列：每个线程只有一个消息队列

  • 基础课没有办法进0环，内核的知识现阶段是漆黑一片，所以只能通过画图的方式来表达
  • 之前先讲的进程和线程，先讲他们是有目的的，不了解进程和线程，就不会了解上面的这幅画面
  • 进程和线程在内核都有一个内核结构体，假设A进程有2个线程，每个线程在内核都有自己的一个大的数组，或者说链表，暂时先不深究，就是相当于一个容器，用来存放消息。也就是说消息是存放到线程里的
  • 消息产生，最先捕获的不是我们的应用程序，比如说一次点击关闭按钮的操作，是操作系统先捕捉到鼠标点击的这个动作，然后再把这个动作发送给对应的进程的某一个对应的线程，线程才会根据这个动作做出相应的处理，也就是关闭窗口
  • 简单说，就是操作系统捕获动作，生成消息结构体，然后把这个结构体放到当前线程的消息队列里，这个消息队列准确的说，是一个复杂的链表，有7个链表，现在暂时不关注
  • 窗口对象，叫winobject，先不管名字，在这个对象里面有一个成员，是个指针，指向了线程。windows通过鼠标位置找到应该属于哪一个窗口，通过遍历窗口对象找到对应的窗口，在通过窗口对象里面的成员找到线程内核对象，在每个线程里面又有一个消息队列
  • 需要说明的一点就是，一个线程可能有很多很多的窗口，但是一个窗口只能对应一个线程

虚拟内存与物理内存

  • 很多新手，也包括一些有工作经验的人，弄不清楚内存相关的一些概念，因为根本没弄清楚什么是虚拟内存，什么是物理内存，也没弄清楚他们有什么关系
  • 操作系统为了便于管理，把物理内存进行了分类。Intel在x86这种架构里面是按4KB进行分配的，也就是把所有的物理内存按照4KB的大小分成一页一页的，这也是为什么称物理内存是页式管理的原因
  • 大家在看内核API的时候，很多时候会看到类似page这种单词，4KB这种大小，因为物理内存一页就是4KB，但也不是绝对的，ARM架构的就是2KB，是由CPU设计的时候要求的

虚拟内存与物理内存关系

  • 现在再说4GB内存空间的概念的时候，要反应过来，是虚拟的4G的内存空间，并没有真正的物理内存，只有真正使用的时候，操作系统才会给挂上一个物理内存，物理内存为了方便管理，使用页式进行管理

  • 我们上面画的物理内存还不是我们插得内存条，中间还有一层映射，但这个过于底层，先不深究

  • 上面空指针赋值区，64K禁入区，在现阶段还不能用，再后面的学习中会，这种传统意义上的不能使用的地方也能使用，想怎么用就怎么用，现在先这么理解

  • 即便用户空间有显示的那么大的一块虚拟内存，但也不会全分配了，可能我们使用到的只有很小的一部分，这就意味着只有我们使用的很小的这一块有对应的物理页，而剩下的没有用到的虚拟内存就没有对应的物理页

物理内存

  • 虚拟内存有4G，那物理内存多大，是不是应该也有一个限制？之前说物理页是以4KB为单位进行管理的，那物理页有多少？谁来决定物理页的多少？
  • 物理内存映射的就是插了一块多大的内存条
  • 可供使用的物理内存
    • NumberOfPhysicalPages * 4 = 物理内存
    • 虚拟内存(硬盘)
  • 能够识别的物理内存
    • 32位系统最多识别物理内存为64G，但由于操作系统限制，比如XP，只能识别4G
  • 虚拟地址使用了，但并不代表挂到它上面的物理页一直都存在，如果这样设计，物理页很快就会占满了。比如A进程使用物理页，操作系统发现用的不频繁了，就会把物理页上的内容存到硬盘上，然后这个物理页就挪作他用了，把它给使用频率高的进程使用了。当A进程又使用的时候，会再调拨一个新的物理页，再把数据从硬盘上拿出来放到这个新的物理页上，对于应用程序来说，这些操作都是透明的，都是感受不到的。

私有内存的申请与释放

  • 我们申请的，所谓的内存只有两种，一种称为私有的，一种称为映射的
  • 私有内存指的是物理页只给某一个进程使用。同一个物理页被多个进程使用共享的时候，这种内存我们称为map的内存

申请内存的两种方式

  • 通过VirtualAlloc/VirtualAllocEx申请的：Private Memory
  • 通过CreateFileMapping映射的：Mapped Memory
  • VirtualAlloc和VirtualAllocEx底层实现是一样的，区别在于VirtualAlloc是在自己的进程里申请内存，VirtualAllocEx可以指定进程id，也就是说可以到其他进程里申请内存
  • 释放内存
  • 真正申请内存，就是上面这两种方式，像之前写的new，malloc跟申请内存没关系，名义上是申请内存，但实际上并不是。maclloc是从已经申请好的内存中，拿出一小块来使用。
  • 可以把VirtualAlloc看成是向操作系统申请批发的过程，是真实存在的向操作系统的申请。malloc是这块内存已经批发下来了，只不过是拿出一小部分来使用。之前讲的堆内存，栈内存，其实在应用程序启动的时候，操作系统已经通过VirtualAlloc这种函数给我们申请好了
  • 到现在再提到栈和堆，应该想到的是，他们是私有内存，是通过vistualloc申请的

文件系统

  • 文件系统和硬盘没有任何关系，仅仅是软件层面的东西

定义

  • 文件系统是操作系统用于管理磁盘上文件的方法和数据结构，简单点说就是在磁盘上如何组织文件的方法

  • 任何东西存到硬盘上的时候都会变成0和1，但是怎么组织，是由文件系统决定的。简单点说就是硬盘负责存0和1，但是怎么存，以什么格式存，则是由文件系统决定的

内存映射文件

  • 内存映射文件就是把硬盘上的一个文件直接映射到物理页上，然后自己接映射到虚拟内存里，这样做的好处就是方便简单，避免了打开关闭哪些步骤，还有一个就是当我们要读写的文件非常大的话，内存映射文件这种方式能够提供非常好的性能，远比io的方式性能好的多
  • flush刷写指的是把数据从物理页写到真正的文件里

内存映射文件之共享

  • 知道了内存映射文件共享，再来看一个知识点
  • 每一个进程通常情况下，都是一个exe带上一堆dll。不管是什么dll，在物理内存上，在物理页上都是只有一份的。比如说上图中画的kernel32.dll，每个进程都会用到操作系统提供的dll，这种dll操作系统不会单独为每个进程分配内存，这样做的话太浪费内存了，之前讲过private和mapping的内存，我们使用的内存绝大多数都是映射的内存，像kernel32.dll这种，通过映射文件的方式把文件映射到物理页上，其他的进程在通过映射的方式映射到虚拟内存里，这样每个进程都会有一种，自己单独有一份这些模块的感觉。
  • 读的时候，共享没有问题，但是写的时候就会出问题，因为大家用的是同一份。这个时候怎么办？这里就用到了写拷贝

写拷贝

  • API
  • 写拷贝，就是写的时候会拷贝一份，不会改原来的物理页，而是复制一份要修改的物理页进行修改
  • 没有修改之前，进程A和进程B指向的都是同一个地方，但是A试图修改的时候，会复制一份进行修改

  • PEB(进程环境块)，存放进程信息，每个进程都有自己的PEB信息，TEB偏移0x30即当前进程的PEB

注入代码

  • 复制代码编写原则
    • 不能有全局变量
    • 不能使用常量字符串
    • 不能使用系统调用
    • 不能嵌套调用其他函数
